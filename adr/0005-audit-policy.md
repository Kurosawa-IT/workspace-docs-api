# ADR0005: 監査ログの粒度・PII除外方針

Status: Accepted  
Date: 2026-02-25

## Context

本プロジェクトはB2B向けのワークスペース型ドキュメント管理SaaSであり、運用上「誰が・いつ・何をしたか」を追跡できることが必須である。
一方で、監査ログに個人情報（PII）や秘匿情報を保存すると情報漏えいリスクが急増し、保管コストも膨らむ。

そのため、監査ログに残す粒度・残さない情報・保持方針を先に固定する。

## Decision

### 1. 監査対象（アクション一覧）

監査ログの `action` は文字列で固定し、以下を最低限の対象とする。

- Document
  - `doc.create`
  - `doc.update`
  - `doc.delete`
  - `doc.publish`
  - `doc.archive`
- Membership（ワークスペースメンバー）
  - `member.add`
  - `member.remove`
  - `member.change_role`
- Export / Jobs（将来）
  - `export.create`
  - `export.download`
  - `job.read`（必要なら）

※ 上記以外も将来追加可能だが、追加時は「PII除外」と「before/after最小化」を守る。

### 2. ログ構造と必須項目

監査ログ（audit_logs）は以下を必須とする。

- `workspace_id`（テナント境界）
- `actor_user_id`（実行者。システムイベントはNULL可）
- `action`
- `target_type` / `target_id`（対象の識別）
- `request_id`（APIログと突合するため）
- `created_at`

任意（取得できる範囲で）：

- `ip` / `user_agent`（運用・不正調査で有用）

### 3. before/after に含める項目（最小）

before/afterは「差分再現」ではなく「意思決定に必要な最小情報」に限定する。

- Document: `title`, `status`, `tags`（本文 body は原則入れない）
- Membership: `user_id`, `role`

理由：

- bodyの全文はPII/機密を含みやすく、容量が大きい
- 監査は「何が変わったかの概要」と「誰がやったか」が重要で、全文の保存は別要件（版管理/履歴）で扱う

### 4. PII/秘匿情報の扱い（基本方針：入れない）

監査ログに以下を保存しない。

- パスワード（平文/ハッシュ問わず）
- トークン（JWT/refresh/api key等）
- 秘密鍵・cookie・セッション識別子
- メールアドレスや氏名などのPII（原則：入れない）
  - どうしても必要な場合は、別テーブル参照可能なIDのみを保存する
  - 例：actor_user_id / target_id を保存し、表示時にJOINで必要最小限を出す

マスクが必要な場合：

- ログに含める前に `***` 等でマスクし、原文を残さない

### 5. 追記のみ（UPDATE/DELETE禁止）

audit_logs は追記のみ（append-only）を運用前提とし、原則 UPDATE/DELETE を禁止する。

理由：

- 監査の信頼性は「後から改変できない」ことに依存する
- 運用上の調査で、ログが欠落/改変されていると説明責任を果たせない

※ 改ざん耐性（ハッシュチェーン等）は将来拡張として別途検討するが、まずは運用ルールで不変性を担保する。

### 6. 保持期間（暫定）とコスト意識

保持期間は暫定で以下とする（プロダクト要件で見直し可能）。

- オンライン保持：90日
- アーカイブ（低頻度アクセス）：1年（圧縮/別ストレージ検討）

コスト対策：

- before/afterは最小化（容量肥大を防ぐ）
- インデックスは検索に必要な最小セットに限定（workspace_id/created_at, workspace_id/actor_user_id/created_at）

## Consequences

メリット:

- 監査の運用が可能になり、不正調査や障害対応がしやすい
- PII漏えいリスクとストレージコストを抑制できる
- アクション命名と構造が固定され、レビューで逸脱を検出しやすい

デメリット/注意点:

- 将来「全文履歴」「差分復元」が必要になった場合は、監査ログとは別に履歴機構を導入する必要がある
- 監査対象が増えると運用コストも増えるため、追加時に目的と粒度を都度確認する
