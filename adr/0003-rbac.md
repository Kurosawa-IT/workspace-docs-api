# ADR0003: RBACモデルと認可集約方針

Status: Accepted  
Date: 2026-02-21

## Context

本プロジェクトは「ワークスペース型（マルチテナント）ドキュメント管理SaaS」を想定している。
APIは常に `workspace_id` スコープで動作し、ユーザーはワークスペースに所属（membership）した上で権限（role）に応じた操作が可能になる。

認可（Authorization）はバグると情報漏えい・権限昇格・監査不能などの重大事故につながるため、
「迷わない」「漏れない」仕組みを先に固定する。

## Decision

### 1. ロール定義（RBAC）

ワークスペース内のロールは以下の4つに固定する。

- owner: ワークスペースの最終責任者。メンバーのrole変更を含む全操作が可能。
- admin: 管理者。メンバー追加/削除は可能だが、role変更は不可（ownerのみ）。
- member: 一般メンバー。ドキュメントの作成/更新/削除など業務操作が可能。メンバー管理は不可。
- viewer: 閲覧者。読み取りのみ可能。

### 2. Actionを文字列で固定し、認可を1箇所に集約する

権限判定は `role × action -> allow/deny` のマトリクスで行う。
Actionは文字列（例: `member.add`, `doc.read`）として固定し、判定は必ず `can(role, action)`（または `authorize()` 相当）を経由する。

ルート実装では以下の形に統一する：

- `Depends(require("<action>"))` を必ず付ける
- `require()` は workspace の所属確認（membership）と action 判定を内部で行う
- 個別ルートで「if role == ...」のような直書きは禁止（漏れ・分岐増殖を防ぐ）

### 3. workspace所属確認と「403 vs 404」の方針

所属していない workspace へのアクセスは **404** を返す方針とする。
理由は以下。

- workspaceの存在有無を推測させない（テナント存在の情報漏えいを避ける）
- ルーティング/実装を単純化でき、レビュー時に判定漏れを検出しやすい

一方、所属はしているが action が許可されない場合は **403** を返す。

- これは「存在は認めるが権限がない」ことを示す。

まとめ：

- 非所属（membership無し）: 404
- 所属あり・権限不足: 403

### 4. 最小アクション一覧（現時点）

- Members: `member.list`, `member.add`, `member.remove`, `member.change_role`
- Documents: `doc.create`, `doc.read`, `doc.update`, `doc.delete`, `doc.publish`, `doc.archive`
- Audit: `audit.read`
- Jobs/Export: `export.create`, `job.read`, `export.download`

## Consequences

メリット:

- 認可の判定が一箇所に集約され、漏れが起きにくい
- ルートの実装パターンが固定され、レビューで「requireが付いているか」だけで検査できる
- 403/404の使い分けが統一され、セキュリティ判断がブレない

デメリット/注意点:

- 将来、リソース単位の細粒度権限（例: docの所有者のみ編集可）を入れる場合は追加設計が必要
- actionが増えるとマトリクスも増えるため、Action命名規約と整理が重要

## Alternatives

- ルートごとに権限を直書きする（if role == ...）
  - 短期は早いが、漏れや分岐増殖で破綻しやすいので不採用
- ABAC（属性ベース）を先に導入する
  - 将来拡張として検討は可能だが、現時点のスコープでは過剰なので不採用
