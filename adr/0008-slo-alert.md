# ADR0008: SLO/アラートの仮置き

Status: Accepted  
Date: 2026-03-08

## Context

本プロジェクトはドキュメント管理・監査・非同期exportを含むAPIであり、障害時に「どこが悪いか」「どの程度ユーザー影響があるか」を早く判断できる必要がある。
一方で、現時点では本番トラフィックや実測値が十分に無く、厳密なSLO/アラート閾値を確定できない。

そのため、初期運用に必要な最低限の監視項目・仮の閾値・通知原則を先に固定し、実測に基づいて後で調整する。

## Decision

### 1. 基本方針

- 初期のSLO/アラートは **仮置き** とする
- まずは「ユーザー影響が大きいもの」を優先する
- ノイズを増やしすぎない（1回の瞬間的な揺れでは鳴らさない）
- 実測値が溜まったら閾値は見直す

### 2. SLO候補 / アラート候補

#### API 5xx率

- 指標:
  - HTTP 5xx rate
- 初期閾値:
  - 5分窓で `> 1%`
- 意味:
  - APIがユーザー要求を正常処理できていない割合
- 優先度:
  - 高

#### API p95 latency

- 指標:
  - HTTP request latency p95
- 初期閾値:
  - 5分窓で `> 500ms`
- 意味:
  - 成功していても体感が悪化している状態
- 優先度:
  - 中〜高

#### Job failed率

- 指標:
  - `failed / (succeeded + failed)`
- 初期閾値:
  - 15分窓で `> 0.5%`
- 意味:
  - export等の非同期処理で恒常的な失敗が増えている
- 優先度:
  - 高

#### Job queue depth

- 指標:
  - queued jobs
- 初期閾値:
  - queued が `> 20` を 10分以上継続
- 意味:
  - worker不足、外部依存遅延、ジョブ滞留
- 優先度:
  - 中

### 3. アラート原則

- 単発のスパイクでは鳴らさず、一定時間継続したときに鳴らす
- まずは「ユーザー影響ベース」で考える
  - 5xx率上昇
  - latency悪化
  - export失敗増加
  - queue滞留
- 開発者都合の細かい内部警告は、初期段階ではアラート化しすぎない
- 1つの障害で複数アラートが大量発火しないよう、できるだけ集約する

### 4. 通知の考え方

- 初期は Slack / メール等の軽量通知を前提とする
- 緊急度は以下で区別する
  - 高: API 5xx率、job failed率
  - 中: p95 latency、queue depth
- 深夜・休日に即時通知する条件は、実測と誤検知率を見て後で調整する

### 5. 今後の見直し

以下が揃ったら本ADRを更新する

- 実トラフィックの分布
- p95/p99の安定値
- job失敗の通常レンジ
- queue depth の通常レンジ
- 誤検知/見逃しの実績

## Consequences

メリット:

- 「何を監視するか」が固定され、運用判断がしやすくなる
- 初期監視の抜け漏れを防げる
- 後で実測ベースに調整する前提が明確になる

デメリット/注意点:

- 仮置き閾値のため、最初は厳しすぎる/緩すぎる可能性がある
- 通知先や当番運用は別途設計が必要
- 指標の意味を運用メンバーが共有していないと、ノイズになる
